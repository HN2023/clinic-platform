Architecture

Five layers, clearly separated.

Each layer has a single responsibility. Changes to billing don't touch scheduling. The database namespace is unreachable from the frontend. Every boundary is enforced — not just documented.

01 Client layer Patient portal, staff dashboard, mobile app, admin panel React
02 API gateway Authentication, routing, rate limiting — single entry point Kong / AWS API GW
03 Core services Scheduling · EHR/EMR · Billing · Notifications Node.js microservices
04 Data & integrations RDS · Redis · S3 · Labs (HL7/FHIR) · Payments AWS managed services
05 Infrastructure Kubernetes · Vault · Grafana · CI/CD pipeline EKS · ArgoCD
Deployment

Zero downtime.
Sub-30-second rollback.

Blue/green deployments with Kubernetes ingress switching. The new version receives 0% of traffic until health checks pass, then 100% switches instantly. The previous version stays warm.

The risk

Deployments during clinic hours

A rolling update that takes 5 minutes to complete leaves patients unable to book appointments mid-deployment. A failed release needs a rebuild, not a fast rollback.

The solution

Atomic traffic switching

Green receives no traffic until fully warmed and smoke-tested. One ingress selector update moves all traffic. Blue stays live as immediate fallback for 30 minutes.

The risk

Secrets in environment variables

Credentials baked into pod specs or Docker images are visible in logs, crash dumps, and to anyone with kubectl access.

The solution

Vault sidecar injection

HashiCorp Vault injects credentials at pod startup. Secrets never touch disk, never appear in source code, and every access is audited automatically.

Key design decisions

DecisionRationale
Blue/green over rolling updatesInstant rollback is non-negotiable for a patient-facing system
Two K8s namespacesNetwork policies enforce pod-level isolation between app and data layers
HPA on gateway + servicesMorning appointment rushes require burst capacity without idle over-provisioning
GitOps via ArgoCDEvery production change is a git commit — full history, easy audit, no manual kubectl in prod
12-stage CI pipelineHIPAA compliance gates run automatically on every PR, not as a periodic audit
HIPAA Compliance

An audit trail that
can't be altered.

Every access to patient data produces a structured, immutable log event. Stored in write-once S3 with KMS encryption — tamper-proof even for infrastructure admins.

What's logged

Full access context

user_id · role · patient_id · action · resource · IP address · timestamp · result. Not "record updated" — exactly who, what, which patient, and when.

Where it goes

Two destinations

Real-time SIEM (Splunk/Datadog) for anomaly detection and alerting. S3 Object Lock (WORM) for 6-year immutable retention required by HIPAA.
